# Issue 報告スキル

現在のセッション中に発生した問題（バグ、想定外の挙動、ハマりどころ、ツールやドキュメントの不備など）を GitHub issue として報告する。報告対象は**このセッションで実際に観測した事象のみ**とし、推測や仮定の問題は含めない。

## 1. コンテキスト収集

以下を Bash ツールで実行し、報告先リポジトリと既存テンプレートを把握する。

- 作業ディレクトリの確認: `pwd`
- git リポジトリか確認: `git rev-parse --is-inside-work-tree 2>/dev/null`
- リモートリポジトリの取得: `git remote get-url origin 2>/dev/null`
- organization / repo の抽出: `git remote get-url origin | sed -E 's|.*github\.com[:/]([^/]+)/([^/.]+)(\.git)?|\1 \2|'`
- issue テンプレート検索:
  ```
  find . -maxdepth 4 \( -path "*/.github/ISSUE_TEMPLATE/*.md" -o -path "*/.github/ISSUE_TEMPLATE/*.yml" -o -name "ISSUE_TEMPLATE.md" -o -name "issue_template.md" \) -not -path "*/node_modules/*" 2>/dev/null
  ```
- 既存の類似 issue の確認（重複報告防止）:
  ```
  gh issue list --limit 20 --state open
  ```

## 2. 前提確認

以下の条件を確認し、問題がある場合は**作業を中断してユーザーに報告**すること。

- git リポジトリでない、または origin が GitHub でない場合は、報告先リポジトリをユーザーに確認する
- `gh` コマンドが未認証の場合は `gh auth login` を案内して中断する
- セッション中に観測した問題がない場合は中断する（推測で issue を作らない）
- 既存 open issue に明らかに同一のものがある場合は、新規作成ではなくその issue を提示して確認する

## 3. 報告対象の整理

セッションの会話履歴を見直し、以下を整理する。

**報告対象に含めるもの:**

- 実際に発生したエラーメッセージ・スタックトレース
- 想定と異なる挙動の具体例（再現手順を含む）
- ツール・ライブラリ・ドキュメントの不備で実害があったもの
- 環境固有の問題（OS, バージョン等が関係する場合）

**報告対象から除外するもの:**

- ユーザーの操作ミスや、すでにこのセッションで解決済みの単純なタイポ
- 観測していない推測ベースの問題
- 個人の好みや軽微な要望（feature request として明示する場合を除く）
- 機密情報（API キー、トークン、個人情報、社内固有のパス等）

## 4. Issue テンプレート読み込み

以下の優先順位でテンプレートを探す。**ファイル名は固定せず、必ず一覧取得してから読み込むこと。**

**優先順位:**

1. **リポジトリのテンプレート**（コンテキスト収集の `find` 結果を使う）
   - `.github/ISSUE_TEMPLATE/*.md` / `.github/ISSUE_TEMPLATE/*.yml`
   - `.github/ISSUE_TEMPLATE.md` / `ISSUE_TEMPLATE.md`
   - **複数ある場合は問題の内容に合うものを自分で選ぶ**（後述）

2. **organization の共通テンプレート**（リポジトリに存在しない場合）

   コンテキスト収集で取得した org 名を使い、`gh api` でディレクトリ一覧を取得してから内容を読み込む。`gh api` は Base64 エンコードで返すので `base64 -d` でデコードする。

   ```bash
   # Step 1: ディレクトリ一覧を取得してファイル名を確認する
   gh api repos/<org>/.github/contents/.github/ISSUE_TEMPLATE 2>/dev/null | jq -r '.[].name'

   # Step 2: 一覧取得できた場合、各ファイルの内容を取得する
   gh api repos/<org>/.github/contents/.github/ISSUE_TEMPLATE/<実際のファイル名> --jq '.content' | base64 -d 2>/dev/null

   # ディレクトリが存在しない場合はルートの ISSUE_TEMPLATE.md を試す
   gh api repos/<org>/.github/contents/.github/ISSUE_TEMPLATE.md --jq '.content' | base64 -d 2>/dev/null
   ```

   **複数ある場合は問題の内容に合うものを自分で選ぶ**（後述）

3. **デフォルト構成**（上記どちらも存在しない場合）

```
## 概要
（何が起きたか・問題の要約）

## 再現手順
1.
2.
3.

## 期待する挙動
（本来こうなるはずだった、という内容）

## 実際の挙動
（実際に観測した挙動・エラーメッセージ）

## 環境
- OS:
- バージョン:
- その他関連情報:

## 補足
（スクリーンショット、ログ、関連 issue 等）
```

**テンプレートが複数ある場合:**

ファイル名・`name` フィールド・`about` フィールドと、報告する問題の内容を照合して最も適切なものを自分で選ぶ。判断基準：

- バグ・不具合・想定外の挙動 → bug report 系
- 新機能・要望 → feature request 系
- ドキュメント不備 → docs 系

どのテンプレートにも当てはまらない、または甲乙つけがたい場合に限りユーザーに確認する。

## 5. Issue プレビュー表示・承認確認

以下の形式で issue の内容をユーザーに提示し、**必ず承認を得てから投稿**すること。承認が得られない場合は中断する。

```
---
タイトル: <英語タイトル>
リポジトリ: <org>/<repo>
ラベル: <該当する場合のみ>

<テンプレートを埋めたボディ>
---
この issue を投稿しますか？
```

- **タイトルは英語**（例: `Crash when running migration with empty config`, `Docs: setup guide missing Node version requirement`）
- **ボディは日本語**でテンプレートを埋めること。**文体は常体（だ・である調）とし、ですます調は使用しないこと**（例: ×「クラッシュします」 → ○「クラッシュする」）
- 再現手順・エラーメッセージは**セッション中に実際に観測したもの**をそのまま使うこと（捏造しない）
- 機密情報（トークン、パス、個人情報等）が含まれていないか目視で確認すること
- 関連する PR / issue がある場合は本文に記載すること（例: `関連: #123`）

## 6. Issue 内容とセッション履歴の整合性確認

ユーザーの承認後、投稿する直前に、issue の内容がセッション中に実際に観測した事実と一致しているか自己レビューする。

**確認観点:**

- 記載した再現手順・エラーメッセージが、セッション中に実際に発生したものか（憶測で補完していないか）
- 実際には観測していない挙動を「実際の挙動」として書いていないか
- 環境情報（OS, バージョン等）はコンテキスト収集で得たものと一致しているか
- 機密情報が混入していないか

**齟齬があった場合:**

issue 本文を書き直し、修正後の内容を**再度ユーザーに提示して承認を得る**こと（step 5 のフォーマットに従う）。承認なしで step 7 に進んではならない。

齟齬がなければそのまま step 7 に進む。

## 7. Issue 投稿

ユーザーの承認を得た後にのみ実行する。

```
gh issue create --title "<英語タイトル>" --body "$(cat <<'EOF'
<テンプレートを埋めたボディ>
EOF
)"
```

ラベルを付ける場合は `--label "<label1>,<label2>"` を追加する。完了後は issue の URL を表示すること。

## 注意事項

- `gh` コマンドが未認証の場合は `gh auth login` を案内して中断すること
- 投稿先リポジトリは現在の作業ディレクトリの origin を既定とする。別リポジトリに投稿したい場合はユーザーに確認すること
- セッション中に観測していない事象を推測で issue 化しないこと
- 機密情報の混入チェックを怠らないこと
