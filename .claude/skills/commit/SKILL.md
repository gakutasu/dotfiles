# コミット作成スキル

現在の変更を Conventional Commits 形式の英語メッセージでコミットする。`git push` は行わない。

## 1. コンテキスト収集

以下を Bash ツールで並列実行し、現在の状態を把握する。

- 現在のブランチ: `git branch --show-current`
- デフォルトブランチの確認: `git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's|refs/remotes/origin/||'`
- ステージング済み・未コミット・未追跡ファイルの確認: `git status --short`
- ステージング済みの差分: `git diff --cached`
- 未ステージングの差分: `git diff`
- 直近のコミット履歴（メッセージのスタイル参考）: `git log --oneline -10`

## 2. 前提確認

以下の条件を確認し、問題がある場合は**作業を中断してユーザーに報告**すること。

- **デフォルトブランチ（`main` / `master`）への直接コミットは禁止**
  - 現在のブランチがデフォルトブランチの場合は中断し、作業ブランチへの切り替えを案内する
- コミット対象の変更が存在しない場合は中断する
- `.env` や認証情報など、秘匿情報を含む可能性のあるファイルがステージングされている場合は警告し、明示的な承認を得る

## 3. コミット範囲の確認

- ステージング済みの変更がある場合: ステージング済みのもののみをコミット対象とする
- ステージング済みの変更がない場合: ユーザーに対象ファイルを確認してから `git add <file>` でステージングする
  - `git add -A` / `git add .` は使用しない（秘匿ファイルや意図しないファイルの混入を防ぐため）

## 4. コミットメッセージ作成

**形式: Conventional Commits（英語、subject のみの 1 行）**

```
<type>: <subject>
```

**body は一切書かない**。`-m` は 1 回のみ使用し、複数行メッセージにしない。

**type の選択肢:**

- `feat` — 新機能の追加
- `fix` — バグ修正
- `docs` — ドキュメントのみの変更
- `style` — コードの動作に影響しない変更（フォーマット等）
- `refactor` — 機能追加・バグ修正を伴わないコード変更
- `perf` — パフォーマンス改善
- `test` — テストの追加・修正
- `build` — ビルドシステムや依存関係の変更
- `ci` — CI 設定の変更
- `chore` — その他の雑多な変更

**メッセージ作成方針:**

- subject は英語の命令形・現在形で記述（例: `Add user authentication`, `Fix null pointer exception`）
- subject の末尾にピリオドを付けない
- subject は 30 文字以内を目安に、簡潔にまとめる
- body は書かない（背景・経緯は PR 本文や issue に記載する）

## 5. コミットプレビュー表示・承認確認

以下の形式でコミット内容をユーザーに提示し、**必ず承認を得てから実行**すること。

```
---
ブランチ: <現在のブランチ>
対象ファイル:
  <ステージング済みファイル一覧>

コミットメッセージ:
<type>: <subject>
---
このコミットを作成しますか？
```

承認が得られない場合は中断する。

## 6. コミット実行

ユーザーの承認後にのみ実行する。メッセージは 1 行のみ。

```
git commit -m "<type>: <subject>"
```

完了後に `git status` で結果を確認する。

## 注意事項

- **`git push` は一切行わない**（プッシュは `/pr` スキルまたはユーザー指示で実行する）
- `--no-verify` などフックをスキップするオプションは使用しない
- `--amend` は使用しない（新しいコミットを作成する）
- pre-commit フックが失敗した場合は原因を調査・修正し、再ステージングして新しいコミットを作成する
